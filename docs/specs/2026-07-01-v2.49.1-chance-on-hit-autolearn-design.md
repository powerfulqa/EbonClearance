# v2.49.1  -  Chance-on-hit proc autolearn

**Status:** approved for implementation
**Date:** 2026-07-01
**Version target:** v2.49.1 (patch  -  additive schema only, no removals)
**Depends on:** v2.49.0 (knownProcPass positive signal, NEVER_EXTRACTABLE hard set, CONFIRMED_ITEMS map)
**Feeds into:** v2.50.0 (peer-share broadcast  -  parked)

## Goal

Grow the chance-on-hit auto-release feature's coverage automatically as the player extracts procs at the PE Anvil, without hand-curating every itemID -> spellID pairing. Each successful extraction observed by the addon widens `playerHasExtractedProc`'s known set for future weapons of the same itemID.

## Why not just hand-curate

The v2.49.0 in-code `EC_CHANCE_PROC_CONFIRMED_ITEMS` map ships with 8 entries. Every weapon PE ports to its 700xxx affix family needs a row, and the community's pool of extractable weapons is much larger than what one person can enumerate. Autolearn removes the manual step: the player extracts once, the addon writes the pairing, and every future drop of that same itemID auto-sells when the toggle is on.

## Architecture

### Data flow

```
BAG_UPDATE ---> EC_ScanLootDelta ---> EC_recentChanceProcRemovals (ring buffer, 5s window)
                                                                 |
LEARNED_SPELL_IN_TAB ---> EC_TryAutolearnFromLearnedSpell(spellID)
                                       |
                                       v
                          [1 removal in window] ---> ADB.chanceProcConfirmedItems[itemID]
                                                    + chat toast
                          [0 or 2+ removals]   ---> ADB.chanceProcAmbiguous (silent)
```

### Ring buffer  -  `EC_recentChanceProcRemovals`

- Populated inside `EC_ScanLootDelta` alongside the existing loot-log write. Same scan pass; no new BAG_UPDATE registration.
- Each entry: `{ itemID, itemName, procLine, removedAt = GetTime() }`.
- Only chance-on-hit items are logged (checked via `EC_compCache.itemHasChanceOnHit`).
- Prune at the start of every scan pass: drop entries where `GetTime() - removedAt > 5`.
- Not persisted across reloads (session-only local).

### `LEARNED_SPELL_IN_TAB` handler

- Registered in `EbonClearance_Events.lua`, alongside the existing event hub.
- Handler snapshot / diff strategy:
  - `EC_extractionCatalogSnapshot` holds `{ [spellID] = true }` for records where `learned == true` at the last event boundary.
  - On event fire, walk `_G.ExtractionService.learnedAffixes`, find records where `learned == true` and `not EC_extractionCatalogSnapshot[record.id]`.
  - If exactly one such record: that is the newly-learned spellID. If zero or multiple: skip (event was ambiguous or unrelated).
  - Refresh `EC_extractionCatalogSnapshot` from the current catalog after every event.
- Handler body:
  1. Diff catalog to find newly-learned spellID (single).
  2. Sanity gate: `spellID >= 700000 and spellID < 800000 and record.weaponOnly == true`. Non-weapon affixes (Iron Will, Frailty stat variants in 900xxx) short-circuit.
  3. Prune the ring buffer of stale entries.
  4. Count non-NEVER_EXTRACTABLE chance-on-hit removals in the buffer:
     - Exactly 1 -> autolearn write + chat toast.
     - 0 or 2+ -> log to `ADB.chanceProcAmbiguous` with `{ spellID, family, timestamp = GetTime(), candidates = { {itemID, itemName, procLine}... } }`.
- Correlation logic isolated in `EC_TryAutolearnFromLearnedSpell(spellID)` so the `/ec autolearnsim` command reuses the same code path.

### `playerHasExtractedProc`  -  new priority order

Extended by one step:

1. `EC_CHANCE_PROC_NEVER_EXTRACTABLE` (in-code hard set) -> return false.
2. `EC_CHANCE_PROC_CONFIRMED_ITEMS` (in-code author-vetted) -> resolve spellID.
3. **`ADB.chanceProcConfirmedItems` (autolearned, per-itemID)** -> resolve spellID. ŌåÉ new
4. `EC_CHANCE_PROC_KEYWORDS` (in-code keyword inference) -> best-effort resolve spellID.
5. Catalog check: `record.id == spellID and record.learned == true` in `_G.ExtractionService.learnedAffixes`.

The two confirmed-items layers merge conceptually (author + autolearn) but stay in separate storage so a `/reload` or wipe of `ADB` doesn't take out the author-vetted pairings.

## New schema

Two new fields on `ADB` (account-wide SavedVariables), initialised by `EnsureAccountDB` with nil-defaults so existing accounts auto-migrate:

```lua
ADB.chanceProcConfirmedItems = ADB.chanceProcConfirmedItems or {}
-- shape: { [itemID] = { spellID, family, item, learnedAt } }

ADB.chanceProcAmbiguous = ADB.chanceProcAmbiguous or {}
-- shape: array of { spellID, family, timestamp, candidates = { { itemID, itemName, procLine } } }
```

`ADB.chanceProcMap` (already reserved in v2.49.0, procLine-keyed, currently empty) stays reserved for the v2.50.0 peer-share transport.

## Slash commands (diagnostics)

### `/ec autolearnsim <itemID> <spellID>`

Simulation harness. Injects a synthetic entry into `EC_recentChanceProcRemovals` for the given `itemID`, then fires `EC_TryAutolearnFromLearnedSpell(spellID)` directly (bypassing the actual `LEARNED_SPELL_IN_TAB` event). Prints the outcome to chat + notes that the run was a simulation.

- Requires the item to be in the player's bags at the moment of the sim so `EC_compCache.chanceProcLine` can read the actual proc text from the live tooltip. If the itemID isn't in any bag slot, the command refuses with a message pointing the player at `/ec captureproc` for a proc-line dump. (`GetItemInfo` alone does not expose the proc line - a bag scan is the only source.)
- Verifies the itemID is a chance-on-hit weapon that isn't in `EC_CHANCE_PROC_NEVER_EXTRACTABLE`; refuses with a message if not.
- Purpose: lets Serv (and any future player) walk the correlation code without needing an actual unlearned weapon proc at the Anvil.

### `/ec autolearnpeek`

Opens a copyable window (same style as `/ec captureproc`, `/ec affixdebug dump`) with three sections:

- **Author-vetted:** every row in `EC_CHANCE_PROC_CONFIRMED_ITEMS`.
- **Autolearned:** every row in `ADB.chanceProcConfirmedItems`, plus timestamp.
- **Ambiguous events:** every entry in `ADB.chanceProcAmbiguous` with all candidate items.

Purpose: read-only inspection surface for debugging + bug reports.

Both commands land in the Main panel's Slash Commands section under the `Diagnostics` heading (added in v2.49.0's section grouping).

## Chat toast

On a successful autolearn write:

```
|cff66ccff[EbonClearance]|r Learned proc pairing: |cffb6ffb6<ItemName>|r extracts to |cffb6ffb6<SpellName>|r. Sell known chance-on-hit procs now covers this item.
```

Localised via `L[]`. One new English string; blank template entries added to `EbonClearance_Locale_frFR.lua` and `_deDE.lua`.

## Sanity gates

Any of these short-circuit the write:

- Learned spellID outside `[700000, 800000)` (PE weapon-proc range).
- Learned record's `weaponOnly ~= true`.
- Zero removals in the ring buffer during the 5-second window (event fired without a correlated bag change).
- 2+ removals in the buffer (ambiguous; can't pick).
- Any candidate itemID in `EC_CHANCE_PROC_NEVER_EXTRACTABLE` (defensive  -  if the hard set says PE can't extract from this item, we don't trust a correlation that says otherwise).

## Non-goals (out of scope for v2.49.1)

- **Peer sharing.** Broadcasting learned pairings to guildmates / party members via `NS.Comms`  -  parked for v2.50.0. `ADB.chanceProcMap` stays reserved for that flow.
- **Manual promotion of ambiguous entries.** Reviewing `ADB.chanceProcAmbiguous` and confirming which candidate was the real extraction is a v2.49.2 feature at earliest; for now, ambiguous events sit silent and the player can inspect via `/ec autolearnpeek`.
- **Auto-delete of unsellable known chance-on-hit weapons.** Deferred to v2.49.2 (separate feature, separate destructive toggle).
- **Auto-delete of grey items on loot.** Deferred to v2.49.2 (separate feature, separate destructive toggle).
- **Retroactive scan.** No walk of the current catalog to backfill from already-learned procs; the initial in-code CONFIRMED_ITEMS map covers the highest-value pairings and future extractions grow it forward. Retroactive backfill is out of scope.

## Ship classification

Patch release. Schema is additive with `EnsureAccountDB` nil-defaults so existing SavedVariables auto-migrate. No removals or breaking changes. Feature-flagged by the existing v2.49.0 `DB.sellChanceOnHitKnown` toggle; when off, autolearn still writes to the tables (they grow silently for future use) but nothing sells.

## Risks

1. **False autolearn via non-extraction removals in the window.** Sanity gate + NEVER_EXTRACTABLE guard reduce this, but a player who sells or destroys a chance-on-hit weapon at the exact moment they extract a stat affix could trigger a bogus write. Mitigation: the 5-second window is tight (typical `LEARNED_SPELL_IN_TAB` fires within 1-2s of the anvil click), and the ambiguous case handles multi-removal correctly. Post-ship: monitor `ADB.chanceProcAmbiguous` growth vs `ADB.chanceProcConfirmedItems` growth in bug reports.
2. **Ring buffer memory growth.** Each entry ~150 bytes, cap of ~10 entries in a 5-second window under normal play. Not a real concern.
3. **`LEARNED_SPELL_IN_TAB` firing without an active PE Anvil session** (e.g. learning a normal spell). Sanity gate on spellID range + `weaponOnly` catches this cleanly.
